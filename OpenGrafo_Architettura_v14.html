2. Panoramica sistema

OpenGrafo è una piattaforma ITSM (IT Service Management) multi-tenant costruita su grafo Neo4j. Ogni tenant ha un sottodominio dedicato (acme.opengrafo.com), un realm Keycloak separato, e dati completamente isolati in Neo4j tramite tenant_id.

Entità ITSM implementate

EntitàWorkflowNote
Incidentnew → assigned → in_progress → escalated/pending → resolved → closedLink a CI, commenti, SLA
ChangeStandard / Normal / Emergency — draft → assessment → planning → CAB → deployment → completedDeployStep, AssessmentTask
Problemopen → in_progress → resolved → closedLink a CI e Incident
ConfigurationItem—Tipi dinamici via CITypeDefinition
WorkflowDefinition—Scope base (condiviso) o tenant
DashboardConfig—Per utente o condiviso nel tenant
AnomalyAlertopen → acknowledged → resolvedScan automatico Neo4j
LogEntry—Filtrato per tenant_id

5. Multi-Tenant via sottodominio

A partire dalla v14, OpenGrafo è progettato per servire N tenant in produzione con completo isolamento. Il tenant slug è l'unica fonte di verità e viene derivato dal sottodominio.

Regole fondamentali

  • Sottodominio = tenant slug = realm Keycloak = tenant_id in Neo4j
  • getTenantSlug() è la sola fonte di verità — nessun hardcode, nessuna env var per il tenant
  • Il sistema funziona per N tenant senza modifiche al codice

Frontend — apps/web/src/lib/keycloak.ts

// Estrae il tenant dal primo segmento dell'hostname
export function getTenantSlug(): string {
  const first = window.location.hostname.split('.')[0]!
  // es. "c-one.localhost" → "c-one"
  // es. "acme.opengrafo.com" → "acme"
  if (first === 'localhost' || first.startsWith('192') ...) {
    throw new Error('Accedi tramite sottodominio: c-one.localhost:5173')
  }
  return first
}

// Keycloak non è più singleton module-level — costruito lazy in initKeycloak()
export async function initKeycloak(): Promise<boolean> {
  const slug = getTenantSlug()
  _keycloak = new Keycloak({
    url:      import.meta.env['VITE_KEYCLOAK_URL'],
    realm:    slug,          // ← realm dinamico dal sottodominio
    clientId: import.meta.env['VITE_KEYCLOAK_CLIENT_ID'],
  })
  ...
}

API — apps/api/src/auth/keycloak.ts

// Cache per-realm — un JwksClient per tenant
const clientCache = new Map<string, JwksClient>()

export async function verifyKeycloakToken(token: string) {
  // 1. Decode senza verifica per estrarre issuer (sicuro — verificato dopo)
  const { iss } = jwt.decode(token) as { iss: string }
  // 2. Estrae realm: "http://kc:8080/realms/c-one" → "c-one"
  const realm = iss.match(/\/realms\/([^/]+)$/)?.[1]
  // 3. Client JWKS per questo realm (creato on-demand)
  const client = getJwksClient(realm)
  // 4. Verifica crittografica con issuer corretto
  return jwt.verify(token, getKey(client), { issuer: iss })
}

Cross-check tenant — apps/api/src/context.ts

6. Migrazione RabbitMQ → BullMQ/Redis

RabbitMQ è stato eliminato completamente. Il sistema usa ora BullMQ su Redis per la gestione degli eventi di dominio.

Fan-out pattern

BullMQ usa code competitive (un solo consumer per coda). Il fan-out si ottiene pubblicando su N code separate — una per consumer:

// packages/events/src/publisher.ts
const CONSUMER_QUEUES = ['notification-service', 'sla-engine'] as const

export async function publish<T>(event: DomainEvent<T>) {
  await Promise.all(
    CONSUMER_QUEUES.map(name => queue(name).add(event.type, event, JOB_OPTIONS))
  )
}

// packages/events/src/consumer.ts
export abstract class BaseConsumer<T> {
  async start() {
    this.worker = new Worker(this.queueName, async (job) => {
      await this.process(job.data as DomainEvent<T>)
    }, {
      connection: getRedisOptions(),
      concurrency: 10,
      settings: { backoffStrategy }  // 5s / 30s / 5min
    })
  }
}

7. Script onboarding tenant

File: apps/api/src/scripts/onboard-tenant.ts

Crea un nuovo tenant completo con un singolo comando. Idempotente — sicuro da eseguire più volte.

Parametri

ParametroObbligatorioDefaultDescrizione
--slugSì—Tenant slug (sottodominio, realm, tenant_id)
--admin-emailSì—Email admin del tenant
--admin-passwordSì—Password iniziale
--admin-first-nameSì—Nome
--admin-last-nameSì—Cognome
--admin-roleNoadminRuolo: admin | user | manager
--domainNoopengrafo.comDominio base produzione

Step eseguiti

  1. Keycloak — Realm: POST /admin/realms con sslRequired: none, enabled: true
  2. Keycloak — Client: opengrafo-web public, redirect URI locali e produzione
  3. Keycloak — Ruoli: admin, user, manager
  4. Keycloak — Mapper JWT: oidc-usermodel-realm-role-mapper → realm_access.roles
  5. Keycloak — Utente: crea utente, imposta password, assegna ruolo
  6. Neo4j — User: MERGE su (email, tenant_id)
  7. Neo4j — DashboardConfig: dashboard default per il tenant
  8. Neo4j — CITypeDefinition: verifica che i tipi base (scope=base) siano disponibili

Esempio: creare tenant "acme" per Acme Corp

pnpm tsx apps/api/src/scripts/onboard-tenant.ts \
  --slug acme \
  --admin-email mario.rossi@acme.com \
  --admin-password Acme1234! \
  --admin-first-name Mario \
  --admin-last-name Rossi

Risultato: http://acme.localhost:5173 e https://acme.opengrafo.com pronti, login con mario.rossi@acme.com / Acme1234!

13. Architettura multi-tenant

Schema a tre livelli

LivelloComponenteResponsabilità
1 — Routing Frontend getTenantSlug() Estrae tenant da window.location.hostname, inizializza Keycloak con realm corretto
2 — Autenticazione Keycloak (realm separato) + API verifyKeycloakToken() Token JWT firmato con chiave del realm tenant. API valida con JWKS del realm estratto da iss
3 — Dati Neo4j + buildContext() Ogni nodo ha tenant_id. Ogni query filtra per $tenantId. Cross-check host/token previene token cross-tenant

Flusso autenticazione completo

Browser su c-one.opengrafo.com
  │
  ▼
getTenantSlug() → "c-one"
  │
  ▼
new Keycloak({ realm: "c-one", ... })
  │
  ▼
Keycloak realm "c-one" → token JWT
  │   iss: "https://opengrafo.com/realms/c-one"
  │
  ▼
API: jwt.decode(token).iss → realm = "c-one"
  │
  ▼
getJwksClient("c-one") → verifica firma
  │
  ▼
getUserByEmail(email) → user.tenantId = "c-one"
  │
  ▼
Cross-check: Host header "c-one.opengrafo.com" = user.tenantId ✓
  │
  ▼
ctx.tenantId = "c-one" → tutte le query Neo4j filtrate

  • Node.js 20 LTS, pnpm 9.x, Docker Desktop

    VITE_KEYCLOAK_URL=http://localhost:8080
VITE_KEYCLOAK_CLIENT_ID=opengrafo-web

# 1. Installa dipendenze
pnpm install

# 2. Avvia infrastruttura
docker compose -f infra/docker-compose.yml up -d
